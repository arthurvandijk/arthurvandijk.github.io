Deep Dive: Glob Patterns in GitHub Copilot Custom Instructions

GitHub Copilot’s custom instructions let you provide project-specific guidance to the AI. By creating a repository file (typically .github/copilot-instructions.md or *.instructions.md in VS Code) you can tailor Copilot’s behavior to your team’s conventions and tools [github-docs]. Organizing these instructions with applyTo filters improves maintainability: each instruction file can automatically apply only to matching files or folders, so you don’t have to duplicate or manually attach guidance. For example, setting applyTo: "**" applies instructions to all files, whereas more specific patterns target only relevant code [github-docs]. This keeps your instructions clean and focused on the files where they matter [github-docs].

Understanding Glob Patterns

Glob patterns are a lightweight syntax for matching file paths. They use wildcards like * and ** to match characters or directory levels.

  • * matches any sequence of characters within a single path segment. For example, *.ts matches all TypeScript files in the current directory [vscode-advanced-search].
  • ** is a recursive wildcard that matches any number of directories (including zero). For example, **/*.md matches all Markdown files in the current directory and any subdirectory [vscode-advanced-search].
  • ? matches exactly one character (e.g. file?.js would match file1.js or fileA.js but not file10.js). Character classes (e.g. [abc]) match any one character in the brackets, and you can use ranges ([A-Z], [0-9]) or negation ([!abc]).

You can combine patterns (e.g., using {...} or comma-separated lists) to match multiple extensions. For example, **/*.{js,ts,tsx} can target several file types at once. Copilot’s applyTo field supports comma-separated patterns, such as "**/*.ts,**/*.tsx", which matches both .ts and .tsx files [github-docs].

Using Glob Patterns in Copilot Custom Instructions

In a Copilot instructions file (a Markdown file with an optional YAML-like header), you can specify an applyTo metadata property to automatically apply the instructions to certain files. For example:

---
applyTo: "**/*.ts,**/*.tsx"
---
Use `PascalCase` for component names and always define Prop interfaces. Prefer React Hooks and functional components for new React files.

Here, the applyTo line tells Copilot to attach these instructions to any TypeScript or TSX file in the workspace [github-docs].

Examples in Real Projects

Frontend (React/TypeScript)

---
applyTo: "**/*.tsx,**/*.ts"
---
**React/TypeScript Guidelines:**
- Use **functional components** and React Hooks (no class components).
- Always define component props via **TypeScript interfaces**.
- Import CSS modules or styled-components for styling (avoid global CSS).
- Use `react-query` or similar for data fetching instead of manual `fetch` calls.

Backend (.NET/C#)

---
applyTo: "**/*.cs"
---
**C# Backend Conventions:**
- Use `PascalCase` for all public class, method, and property names.
- Handle async I/O with `async`/`await`; do not use `.Result` or `.Wait()`.
- Always use the `ILogger` interface for logging (do not write to console directly).
- Follow the company’s DI framework for service registration.
- Validate all web request models with Data Annotations.

Web API (Node.js/TypeScript)

---
applyTo: "routes/**/*.js,src/api/**/*.ts"
---
**API Project Guidelines:**
- Use `express.Router()` for defining endpoints (organize routes by feature).
- Always return appropriate HTTP status codes.
- Validate request bodies with a schema (e.g. use `Joi` or `ajv`).
- Include descriptive comments for each endpoint function.
- Log errors using a centralized middleware.

Unit Test Projects

---
applyTo: "**/*.test.js,**/*.spec.ts,**/tests/**/*.ts"
---
**Unit Test Guidelines:**
- Use descriptive test case names.
- Structure tests with Arrange/Act/Assert comments or blocks.
- Always reset mocks/spies in an `afterEach` block.
- For C#: Place xUnit tests under a `Tests/` folder and use `[Fact]`/`[Theory]`.
- Aim for at least 80% coverage on new features.

Summary

Glob patterns offer a powerful way to scope Copilot’s custom instructions to the right parts of your codebase. By leveraging these filters and organizing instruction files carefully, teams can maintain clean, focused AI guidance that evolves with the project.
